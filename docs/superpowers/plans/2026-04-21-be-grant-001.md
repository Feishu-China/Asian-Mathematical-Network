# BE-GRANT-001 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the minimum real backend slice for conference-linked travel grants: persisted `GrantOpportunity` objects, public grant read APIs, applicant grant draft/create/read/update/submit APIs, and prerequisite validation that requires an already submitted conference application for the linked conference.

**Architecture:** Extend the existing SQLite Prisma workflow schema instead of inventing a parallel grant storage layer. Keep grant opportunities as their own model, keep `grant_application` in the shared `Application` table, add grant-specific serializers/controllers/routes, and generalize `/api/v1/me/applications/:id/draft` plus `/submit` so they dispatch by application subtype without regressing the existing conference flow. Keep transport payloads in spec-aligned `snake_case`. Do **not** implement organizer-side grant management, decision release, or post-visit report in this feature.

**Tech Stack:** Express 5, Prisma 5, SQLite, Jest, Supertest, TypeScript

- [ ] **Step 3: Extend Prisma schema for grant persistence**

Modify [`backend/prisma/schema.prisma`](/Users/brenda/Projects/Asian-Mathematical-Network/backend/prisma/schema.prisma) to add:

- `GrantOpportunity`
- `User.createdGrantOpportunities`
- `Conference.linkedGrants`
- `Application.grant`
- grant uniqueness/indexes on `Application`

Recommended target shape:

```prisma
model User {
  // existing fields...
  createdGrantOpportunities GrantOpportunity[] @relation("GrantOpportunityCreator")
}

model Conference {
  // existing fields...
  linkedGrants GrantOpportunity[] @relation("GrantOpportunityLinkedConference")
}

model GrantOpportunity {
  id                        String        @id @default(uuid())
  linkedConferenceId        String
  slug                      String        @unique
  title                     String
  grantType                 String
  description               String?
  eligibilitySummary        String?
  coverageSummary           String?
  applicationDeadline       DateTime?
  status                    String        @default("draft")
  reportRequired            Boolean       @default(false)
  applicationFormSchemaJson String        @default("{\"fields\":[]}")
  settingsJson              String        @default("{}")
  publishedAt               DateTime?
  closedAt                  DateTime?
  createdByUserId           String
  createdAt                 DateTime      @default(now())
  updatedAt                 DateTime      @updatedAt
  linkedConference          Conference    @relation("GrantOpportunityLinkedConference", fields: [linkedConferenceId], references: [id], onDelete: Cascade)
  creator                   User          @relation("GrantOpportunityCreator", fields: [createdByUserId], references: [id], onDelete: Cascade)
  applications              Application[]

  @@index([status, applicationDeadline])
  @@index([linkedConferenceId, status])
}

model Application {
  // existing fields...
  grant   GrantOpportunity? @relation(fields: [grantId], references: [id], onDelete: Cascade)

  @@unique([conferenceId, applicantUserId, applicationType])
  @@unique([grantId, applicantUserId, applicationType])
  @@index([conferenceId, status])
  @@index([grantId, status])
  @@index([linkedConferenceApplicationId])
  @@index([applicantUserId, status])
}
```

Important:

- The existing `conferenceId` uniqueness is not enough for grants because `conferenceId` stays `NULL` for `grant_application`.
- SQLite migration may need to recreate the `Application` table to add the new foreign key/index layout cleanly.
