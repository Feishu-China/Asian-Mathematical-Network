# INT-GRANT-001 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Verify the real `GRANT` flow end-to-end against the real backend and real frontend runtime wiring: a user can discover a published grant, is blocked from creating a grant draft before submitting the linked conference application, can create/edit/submit a separate `grant_application` after the conference prerequisite is met, and existing `CONF` submit/hydrate behavior remains intact.

**Architecture:** Treat `INT-GRANT-001` as a real-service verification feature, not a new product slice. `BE-GRANT-001` and `FE-GRANT-001` already exist; this feature should only add the minimum glue and verification needed to prove they work together in the actual runtime path. Because organizer-side grant CRUD is still out of scope, deterministic integration data must be seeded directly through Prisma-backed helpers instead of inventing new public management APIs. Only make code changes where the integration run reveals a real mismatch.

**Tech Stack:** Express 5, Prisma 5, SQLite, React, Vite, Axios, Jest, Vitest, local dev servers, browser/runtime verification

## Implementation Notes

- The critical integration risk is not grant form rendering; it is the conference-to-grant dependency rule in the real runtime path.
- Because organizer grant CRUD does not exist, deterministic DB seeding is the cleanest way to make integration reproducible without expanding product scope.
- If the integration run succeeds without any code fixes beyond the fixture/runner, that is acceptable; `INT-GRANT-001` is primarily a proof feature.
- If the integration run uncovers only a small runtime mismatch, fix the narrowest file that owns the mismatch instead of refactoring adjacent modules.

- a reproducible real-service check proves the published grant can be discovered and applied for
- the prerequisite failure path is verified against the real backend
- the success path after conference submission is verified against the real backend
- focused frontend grant + conference regressions pass
